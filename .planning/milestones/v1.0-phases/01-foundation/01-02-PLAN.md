  <action>
    1. Run the following ffprobe commands on the user-provided file (substitute the actual path):
       ```bash
       # Full format and stream JSON
       ffprobe -v error -show_format -show_streams -print_format json /path/to/sample.webp

       # Quick container format name
       ffprobe -v error -show_entries format=format_name -print_format default=noprint_wrappers=1 /path/to/sample.webp

       # Audio codec
       ffprobe -v error -show_entries stream=codec_name,codec_type -print_format default=noprint_wrappers=1 /path/to/sample.webp

       # Magic bytes (first 12 bytes)
       xxd /path/to/sample.webp | head -2
       ```

    2. Interpret the `format_name` value:
       - `webm` or `matroska,webm` -> File is a WebM container with `.webp` extension
       - `webp` -> File is a genuine WebP container (unexpected — investigate audio stream presence)
       - `riff` -> Non-standard RIFF container (unexpected — requires deeper investigation)

    3. Record the `codec_name` from the audio stream:
       - `opus` -> Use `-c:a libopus` in Phase 4 trim commands
       - `vorbis` -> Use `-c:a libvorbis` in Phase 4 trim commands

    4. Test AudioContext.decodeAudioData by adding a temporary test to the running app:

       Or, create a temporary file input in App.tsx that reads the file and attempts decodeAudioData.
       Document whether the browser can natively decode the file (affects Phase 2 waveform strategy).
